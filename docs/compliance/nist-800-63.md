# NIST SP 800-63 — Vouch Protocol™ mapping

**Status:** Skeleton. The mappings below are draft and require legal
review for any production compliance claim.

NIST SP 800-63 defines three Assurance Levels for digital identity:
**IAL** (Identity Assurance), **AAL** (Authentication Assurance), and
**FAL** (Federation Assurance). Vouch Protocol most directly addresses
**AAL** and **FAL** — proving *the right entity authenticated* and
*federated identity assertions are protected* — for autonomous agents
acting on behalf of human principals.

## Summary

**Where Vouch helps:**
- **AAL2 / AAL3:** Per-action signing with Ed25519 cryptographic
  authenticators meets AAL2 in principle. With hardware-isolated keys
  (Identity Sidecar deployed with HSM or Secure Enclave), AAL3 is
  achievable.
- **FAL2 / FAL3:** Vouch Credentials are signed assertions about
  identity and intent. Federated assertion protection requires the
  assertion to be encrypted in transit or signed-and-verified — both
  are protocol-supported.
- **Authenticator binding** (SP 800-63B §5): The agent's signing key
  is bound to the agent's DID at issuance time; the binding is
  cryptographically verifiable through the DID Document.

**Where Vouch does NOT address:**
- **IAL** (Identity Assurance) at any level. IAL concerns *how
  confident we are that an identity corresponds to a real-world
  entity*, which is upstream of any cryptographic protocol. A separate
  identity-proofing process (the kind Vouched.id and similar IDV
  vendors provide) is required to establish the human principal at the
  root of the delegation chain.

**Minimum conformance level recommended:**
- L2 for AAL2 / FAL2
- L3 for AAL3 / FAL3 (PQ readiness + continuous attestation)

## Requirements mapping (selected, not exhaustive)

| Source | Requirement (paraphrased) | Vouch mechanism | Spec section |
|---|---|---|---|
| 800-63B §4.2.2 | AAL2 multi-factor authentication | Vouch Credential = something-you-have (the key) + intent binding | §5, §10 (Sidecar) |
| 800-63B §4.3.2 | AAL3 hardware-based authenticator | Identity Sidecar deployed against HSM or TPM | §10.4 |
| 800-63B §5.1 | Authenticator strength | Ed25519 (default), ML-DSA-44 (PQ); FIPS 186-5 / FIPS 204 | §6, §13 |
| 800-63B §6.2.4 | Reauthentication for AAL2 (≤ 30 minute idle) | Heartbeat Protocol (60s default) + Trust Entropy decay enforces continuous reauthentication | §15 |
| 800-63C §4.2 | FAL2 assertion protection | Data Integrity proof; replay nonce; intent binding | §5, §6 |
| 800-63C §5.3 | Assertion replay resistance | UUID `id` + nonce store; bound `target` and `resource` fields | §5.6 |

## Deployment checklist

- [ ] AAL2 / FAL2: deploy at conformance Level 2 (Sidecar + delegation + revocation)
- [ ] AAL3 / FAL3: deploy at conformance Level 3 with HSM or Secure Enclave for key storage
- [ ] Implement reauthentication via Heartbeat Protocol at ≤ 30-minute equivalent decay (set λ accordingly)
- [ ] Configure ML-DSA-44 dual-proof for any deployment that needs post-quantum AAL claims
- [ ] Pair Vouch (AAL/FAL) with an upstream identity-proofing process for IAL

## Open questions / gaps

- **IAL boundary:** Vouch does not assert IAL. Deployments must combine
  Vouch with an IAL-1/2/3 identity-proofing process. The output of
  identity-proofing populates the root principal at the top of the
  Vouch delegation chain.
- **Phishing-resistant authentication (800-63B §4.2.2 update 2023):**
  Vouch with hardware-isolated keys (Sidecar against HSM) is
  phishing-resistant by construction; software-key deployments are not.
- **Step-up authentication:** Trust Entropy decay provides a
  cryptographic equivalent of step-up: high-stakes actions require a
  fresh heartbeat (higher trust threshold), low-stakes actions accept
  decayed trust. Mapping to 800-63B's "transactional authentication"
  language is a deployment-level explanation.
