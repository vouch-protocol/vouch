    <h2>How It Works</h2>

    <p>Vouch leverages <strong>SSH signing</strong> (introduced in Git 2.34+). When you run
      <code>vouch git init</code>, it:
    </p>

    <ol>
      <li>Generates a new Ed25519 signing key (or uses your existing one at <code>~/.ssh/id_ed25519</code>)
      </li>
      <li>Uploads the key to GitHub as a signing key (via <code>gh</code> CLI)
      </li>
      <li>Configures Git to sign all commits:
        <pre><code>git config --global gpg.format ssh
git config --global user.signingkey ~/.vouch/keys/vouch_signing_key
git config --global commit.gpgsign true</code></pre>
      </li>
      <li>Sets up local verification with an allowed signers file</li>
    </ol>

    <h2>GitHub Integration (Automatic!)</h2>

    <p><code>vouch git init</code> <strong>automatically uploads your signing key to GitHub</strong> using the
      GitHub CLI. You don't need to do anything manually!</p>

    <p>After running the command, you'll see:</p>
    <pre><code> Uploading signing key to GitHub...
   SSH key uploaded to GitHub!</code></pre>

    <p>That's it! Push a commit and you'll see the Verified badge immediately.</p>

    <div class="vouch-blog-callout"><strong>Don't have GitHub CLI?</strong> If <code>gh</code> isn't installed, Vouch will open your browser
      to the GitHub SSH settings page and show you the key to paste. But we recommend installing
      <a href="https://cli.github.com/">GitHub CLI</a> for a fully automated
      experience.
    </div>

    <h2>Manual Configuration (Alternative)</h2>

    <p>If you prefer to configure git manually without the CLI:</p>

    <pre><code># Use SSH for signing
git config --global gpg.format ssh

# Set your signing key
git config --global user.signingkey ~/.ssh/id_ed25519.pub

# Auto-sign all commits
git config --global commit.gpgsign true

# Auto-sign all tags
git config --global tag.gpgsign true</code></pre>

    <h2>Troubleshooting</h2>

    <h3>Commit fails with "error: Load key failed"</h3>
    <p>Your SSH key might require a passphrase. Add it to your SSH agent:</p>
    <pre><code>eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519</code></pre>

    <h3>GitHub doesn't show "Verified"</h3>
    <p>This usually means the <code>gh</code> CLI upload failed silently. Check if your key is on GitHub:</p>
    <pre><code>gh ssh-key list</code></pre>
    <p>If not listed, re-run <code>vouch git init</code> or manually add the key at GitHub Settings → SSH Keys.</p>

    <h3>Check your current git signing config</h3>
    <pre><code>git config --global --list | grep -E "(gpg|sign)"</code></pre>

    <h2>How Others Verify Your Commits</h2>

    <p>When you sign commits, here's how collaborators and reviewers can verify they're genuinely from you:</p>

    <h3>1. GitHub's Verified Badge (Automatic)</h3>
    <p>When you push signed commits to GitHub:</p>
    <ul>
      <li>GitHub checks if the signing key matches a key registered to your account</li>
      <li>If it matches → <strong>Verified</strong> badge appears on the commit</li>
      <li>Click the badge to see: <em>"This commit was signed with the committer's verified signature"</em></li>
    </ul>

    <h3>2. Verify Anyone's Public Key</h3>
    <p>Anyone can check a GitHub user's public SSH keys at:</p>
    <pre><code>https://github.com/&lt;username&gt;.keys</code></pre>
    <p>Compare this with the key fingerprint shown in the commit signature to confirm authenticity.</p>

    <h3>3. Local Verification</h3>
    <p>To verify signatures locally:</p>
    <pre><code># View signature details
git log --show-signature -1 &lt;commit-hash&gt;

# Verify a specific commit
git verify-commit &lt;commit-hash&gt;</code></pre>

    <div class="vouch-blog-callout"><strong>Key Point:</strong> The SSH key fingerprint in the signature can be matched against the author's
      publicly listed keys on GitHub, providing cryptographic proof of authorship.
    </div>

    <h2>For Teams & Organizations</h2>

    <p>Adopting signed commits across your team provides:</p>

    <ul>
      <li><strong>Supply Chain Security:</strong> Every commit is cryptographically tied to a verified developer
      </li>
      <li><strong>Audit Trail:</strong> Immutable proof of who authored each change</li>
      <li><strong>Compliance:</strong> Meet SOC2, ISO 27001, and other security standards requiring code

      <li><strong>Trust:</strong> Reviewers can verify PRs come from legitimate team members</li>

    <ol>
      <li>Go to <strong>Repository Settings → Branches</strong></li>
      <li>Add a branch protection rule for <code>main</code></li>
      <li>Enable <strong>"Require signed commits"</strong></li>
    </ol>

    <p>Now all commits to protected branches must be signed!</p>

    <h2>Why This Matters</h2>

    <p>Recent supply chain attacks highlight why commit signing is critical:</p>

    <ul>
      <li><strong>SolarWinds (2020):</strong> Attackers injected malicious code into build pipelines</li>
      <li><strong>Codecov (2021):</strong> Compromised CI scripts exfiltrated credentials</li>
      <li><strong>XZ Utils (2024):</strong> Maintainer impersonation led to backdoored releases</li>
    </ul>

    <p>Signed commits create a <strong>chain of custody</strong>. If every commit is signed, attackers can't inject
      code without either stealing a developer's private key or creating obviously unsigned commits.</p>

    <div class="vouch-blog-callout"><strong>By the Numbers:</strong> GitHub reports that repositories with signed commits enabled see
      significantly fewer successful supply chain attacks. The Verified badge is becoming a trust signal
      that security-conscious organizations look for.
