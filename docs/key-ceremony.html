Key Ceremony Documentation

Vouch Protocol C2PA Certificate Authority

Version 2.0 March 2026

1. Purpose

This document describes the key ceremony procedures for the Vouch Protocol C2PA Certificate Authority (CA). The key ceremony is the process by which the CA root key, intermediate signing keys, and leaf certificates are generated, secured, and activated.

CA certificates (Root and Intermediate) use ECDSA P-384 (secp384r1) as required by the C2PA Certificate Policy. Leaf signing certificates use Ed25519 for high-performance content credential signing. All CA key material is generated and stored in Google Cloud KMS HSM (FIPS 140-2 Level 3).

2. Trust Chain Architecture

The Vouch Protocol C2PA CA operates a three-tier certificate hierarchy:

  • Vouch Protocol Root CA (ECDSA P-384) - Keys in Google Cloud KMS HSM, non-exportable protection
  • Vouch Protocol Intermediate CA (ECDSA P-384) - Signs leaf certificates, keys in Google Cloud KMS HSM
  • Vouch Protocol C2PA Signer (Ed25519) - Active signing certificate for content credentials

The root CA key is generated in HSM and never leaves the hardware boundary. The intermediate CA key signs leaf certificates. Leaf certificates are used for day-to-day C2PA content credential signing by subscribers such as Cygn (cygn.me).

3. Key Generation Procedure

The following procedure is followed for each key generation ceremony:

Step 1 Environment Preparation

A clean environment is prepared with Google Cloud SDK, OpenSSL 3.x, and the Vouch Protocol ceremony script. Authenticated access to Google Cloud KMS is verified. All previous temporary files are purged. Network connectivity to Google Cloud KMS API is confirmed.

Step 2 Ceremony Participants

Minimum 3 participants assemble: (1) CA Administrator who executes the ceremony script, (2) Witness who observes and verifies each step, (3) Independent Witness who is not affiliated with the CA Operator. All participants sign the ceremony attestation document.

Step 3 Root CA Generation (One-time)

An ECDSA P-384 asymmetric signing key is created in Google Cloud KMS with HSM protection level and DESTROY scheduling. The public key is extracted via KMS API. A self-signed root certificate is assembled with 20-year validity, basicConstraints CA:TRUE pathlen:2, keyUsage keyCertSign+cRLSign, and C2PA Certificate Policy OID 1.3.6.1.4.1.62558.1.1. The certificate TBS is signed via KMS asymmetricSign API.

Step 4 Intermediate CA Generation

An ECDSA P-384 key is created in KMS HSM. A certificate is assembled with 1,827-day (5-year) validity, basicConstraints CA:TRUE pathlen:0, extendedKeyUsage c2pa-kp-claimSigning (1.3.6.1.4.1.62558.2.1) + emailProtection. AIA extension includes OCSP responder URL and CA Issuers URL. CRL Distribution Point is set. The certificate is signed by the Root CA via KMS API.

Step 5 Leaf Certificate Generation

An Ed25519 keypair is generated locally or in HSM. A certificate is assembled with 366-day (AL1) or 90-day (AL2) validity, basicConstraints CA:FALSE, keyUsage digitalSignature+nonRepudiation, extendedKeyUsage c2pa-kp-claimSigning + emailProtection. C2PA Assurance Level extension (1.3.6.1.4.1.62558.3) and CPL Record extension (1.3.6.1.4.1.62558.4) are included. Signed by Intermediate CA via KMS API.

Step 6 PKI Linting

All generated certificates are validated using PKI linting tools (zlint, x509lint, certlint). C2PA-specific extensions are verified for correct OIDs, criticality flags, and encoding. Results are recorded in the ceremony attestation.

Step 7 Chain Verification

The full certificate chain is verified: Root CA self-signature, Intermediate signed by Root, Leaf signed by Intermediate. Each certificate's extensions (basicConstraints, keyUsage, EKU, AIA, CRL, C2PA OIDs) are independently checked. A test C2PA content credential is signed and verified end-to-end.

Step 8 Sign-off

All three participants review and sign the ceremony attestation document. The attestation includes certificate fingerprints, key IDs, verification results, and any deviations. Signed attestation is archived with 7-year retention.

4. Key Storage and Protection

  • Root CA private key: Google Cloud KMS HSM, FIPS 140-2 Level 3, non-exportable, EC_SIGN_P384_SHA384
  • Intermediate CA private key: Google Cloud KMS HSM, FIPS 140-2 Level 3, non-exportable, used only via KMS API
  • Leaf private key: Subscriber-managed storage (environment variables, local keystore, or HSM)
  • Certificate chain: Published at /.well-known/c2pa-trust-list

All KMS operations require IAM authentication. Key access is restricted to the ceremony service account and authorized signing services. Dual control is enforced for key management operations.

5. Certificate Lifecycle

  • Root CA: 20-year validity, offline renewal ceremony with independent witness
  • Intermediate CA: Maximum 1,827 days (5 years), signed by Root during key ceremony
  • Leaf AL1: Maximum 366 days, annual renewal
  • Leaf AL2: Maximum 90 days, quarterly renewal
  • Re-authentication: Maximum 398 days for certificate renewal

Revoked certificates are published in the CRL at /api/v1/c2pa/crl and queryable via OCSP at /api/v1/c2pa/ocsp.

6. Revocation Procedures

In the event of key compromise or certificate misuse:

Detection

Key compromise is detected through KMS audit log monitoring, anomalous signing patterns, or external reports to ca@vouch-protocol.com.

Immediate Response

The compromised certificate serial number is added to the revocation list. CRL and OCSP endpoints immediately reflect the revocation. For KMS-hosted keys, the key version is disabled via Cloud KMS API.

Key Rotation

A new leaf certificate is generated using the existing intermediate CA via KMS API signing. The new certificate chain is distributed to the subscriber. The old leaf private key is securely deleted.

Notification

Affected subscribers are notified via email. The trust list at /.well-known/c2pa-trust-list is updated with the new certificate information.

7. Audit Controls

  • Google Cloud KMS provides built-in audit logging for all key operations
  • KMS audit logs are exported to Cloud Logging with 7-year retention
  • Ceremony artifacts (attestations, PKI lint reports, chain verifications) retained for 7 years
  • Certificate issuance and revocation events logged with operator identity and timestamp
  • CRL and OCSP query logs maintained for compliance reporting
  • C2PA signing operations logged per subscriber with content hash and certificate serial number

8. Compliance

The Vouch Protocol C2PA CA operates in accordance with:

  • C2PA Certificate Policy (OID: 1.3.6.1.4.1.62558.1.1)
  • C2PA Specification v2.x - Content Credentials trust model
  • FIPS 140-2 Level 3 - Hardware security module certification (Google Cloud KMS)
  • FIPS 186-4 - Digital Signature Standard (ECDSA P-384)
  • RFC 5280 - Internet X.509 PKI Certificate and CRL Profile
  • RFC 6960 - Online Certificate Status Protocol (OCSP)
  • RFC 8032 - Edwards-Curve Digital Signature Algorithm (Ed25519)
  • RFC 3161 - Internet X.509 PKI Time-Stamp Protocol (via DigiCert TSA)

Vouch Protocol is a member of C2PA, W3C, and the Linux Foundation.
