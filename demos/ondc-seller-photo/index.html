Vouch × ONDC Live demo

A seller listing that any buyer app can verify, offline.

On ONDC, a single seller listing reaches buyers through dozens of buyer apps the seller never sees, never authenticates with, and has no shared session with. Today, those buyer apps can only trust the listing's claims as far as they trust the seller app it came through. This demo shows what changes when every listing carries its own portable cryptographic proof.

The Problem

A seller posts a saree on a Seller App. The listing flows down the ONDC pipe, gets cached by a few buyer apps, gets indexed by aggregators, gets shown to a buyer. The buyer's app rendered a photo and a title and a price. Three questions it cannot answer cryptographically:

  • Did this seller actually publish that photo, or did someone in the pipeline swap it for a stolen one?
  • Is the title the seller wrote, or did an intermediary adjust it?
  • If the buyer disputes the order later, what does the seller's signed version of this listing look like?
Today the answer is "we trust the seller app's API" — a network-level trust. What ONDC needs is per-artefact trust: a portable proof that travels with each listing wherever it goes.

How To Demo

  1. Open the Seller App. It generates your seller DID (did:web:vouch-protocol.com:u:seller-...) on first run.
  2. Pick any image from your laptop. Set a listing ID and title (defaults are fine). Click Sign listing with Vouch.
  3. Copy the bundle JSON. Open the Buyer App in another tab. Paste it. Drop the same image. Click Verify offline.
  4. Watch the six-step verification pass: signer DID resolved → public key extracted → photo hash matches → signature verifies.
  5. Try the three tamper buttons. Each one breaks a different layer: the photo (hash mismatch), the listing title (canonicalization mismatch), the signature (Ed25519 verification fails).

Why It's Different From "Just C2PA" or "Just Signed URLs"

  • Portable identity — the signer is a DID, not a domain or an API key. The seller can move buyer apps, change hosting, drop a vendor; the identity attached to past listings stays valid.
  • Self-describing intent — the credential carries the listing ID, title, seller handle, and the photo hash, all under one signature. Tamper with any one and the whole proof fails.
  • Verifiable offline — the buyer app needs the credential + the DID Document + the photo. None of those require a live network call. Important for low-bandwidth buyer apps, kiosks, and disputes where the seller has gone offline.
  • Network-neutral — works regardless of which buyer app, which logistics partner, which gateway. The proof travels with the artefact.

What This Buys ONDC

  • Dispute resolution with cryptographic ground truth. When a buyer claims "the photo I saw isn't the product I got", there is a signed canonical version to compare against. The current ONDC dispute flow has no equivalent.
  • Image-theft attribution. When seller A's photos appear on seller B's listing, the signed timestamp on A's credential is a tie-breaker no buyer app or logistics partner can manipulate.
  • Cross-app catalog integrity. Buyer apps and aggregators that re-index ONDC catalogs can verify each listing once at ingestion and serve a "✓ Verified" badge, with zero ongoing trust in the seller app.
  • Network-level audit. Regulators or the ONDC technical office can verify any disputed listing without depending on either party's logs.

What's Next, If This Lands

  • Bind the credential to ONDC's existing provider.id and item.id fields so it slots into the current schema without breaking buyer apps.
  • Extend to other artefacts: KYC documents, GST certificates, return-eligibility certifications, delivery acknowledgements.
  • Reverse the flow: have buyer apps issue Vouch credentials for received-order acknowledgements, signed with the buyer's identity, for closing the loop on "did the order arrive as described".
