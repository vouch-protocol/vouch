SUBSYSTEM SUMMARY / S.01

What the protocol provides

Six properties that traditional API keys, OAuth tokens, and bearer credentials cannot provide for autonomous AI agents operating in regulated environments.

[01]

Cryptographic Identity

Every agent holds a Decentralized Identifier (did:web, did:key) backed by an Ed25519 keypair. No shared secrets.

REF §6 / IDENTITY MODEL
[02]

Intent Attestation

Every action is signed as a VC with Data Integrity proofs binding identity, action, target, and resource.

REF §5 / CREDENTIAL FORMAT
[03]

Resource Delegation

Multi-agent systems gain verifiable principal-to-sub-agent chains. Each link narrows the resource scope.

REF §9 / DELEGATION CHAINS
[04]

Identity Sidecar

Private keys never enter the LLM context window. A separate Go binary signs over a local IPC channel.

REF §10 / IDENTITY SIDECAR
[05]

Heartbeat Renewal

Long-running agents renew SessionVoucher credentials on a periodic schedule. "Untrusted until renewed."

[04.02]
Do I need to run the Go sidecar? [-]

Only if you want the Identity Sidecar pattern, which is the recommended deployment for LLM-driven agents because it keeps the private key out of the LLM context window. The sidecar is a long-running Go daemon that exposes POST /sign on a configurable port.

If your code never exposes the key to the model, you can sign directly from the Python or TypeScript SDK using Signer.sign() or signer.sign().
