Cryptographic identity & accountability for autonomous AI agents.

The Vouch Protocol is an open specification for establishing continuous state verifiability of autonomous AI agents, a layer that sits beneath, and complements, agent identity and delegation specifications. Built on Verifiable Credentials, Data Integrity proofs, and Decentralized Identifiers, with reference implementations in Python, TypeScript, and Go.

§ I

What the protocol provides

i.

Cryptographic Agent Identity

Every agent holds a Decentralized Identifier (did:web or did:key) backed by an Ed25519 keypair. No API keys, no shared secrets, no bearer tokens.

§6 Identity Model
ii.

Intent-bound Credentials

Every action is signed as a Verifiable Credential with Data Integrity proofs. The credential binds identity, action, target, and resource so nothing replays elsewhere.

§5 Credential Format
iii.

Resource-scoped Delegation

Multi-agent systems gain verifiable principal-to-sub-agent chains. Each link narrows the resource scope. No more "the agent did it" black boxes.

§9 Delegation Chains
iv.

Identity Sidecar Pattern

Private keys never enter the LLM context window. A separate Go binary (vouch-sidecar) signs on behalf of the agent over a local IPC channel.

§10 Identity Sidecar
v.

Continuous Trust via Heartbeat

Long-running agents renew SessionVoucher credentials on a periodic schedule. The trust model inverts from "trusted until revoked" to "untrusted until renewed."

§11 Heartbeat Protocol
vi.

Hybrid Post-Quantum Ready

The cryptosuite hybrid-eddsa-mldsa44-jcs-2026 ships today. Ed25519 + ML-DSA-44 sign the same JCS bytes.

§13 Crypto-Agility

About Vouch

What is the Vouch Protocol?

i.
What problem does Vouch Protocol solve?

Autonomous AI agents are increasingly deployed to take real-world actions in regulated environments, financial trades, healthcare claim submissions, regulatory filings. Existing authentication mechanisms (API keys, OAuth bearer tokens, session cookies) were designed for human users and stateless services and do not provide cryptographic chain-of-custody, intent attestation, or behavioral attestation that regulated environments require.

Vouch Protocol fills this gap using primitives already familiar to community: Decentralized Identifiers, Verifiable Credentials, and Data Integrity proofs.

Cf. §1.1 Problem Statement.
ii.
How is this different from API keys or OAuth?

API keys and OAuth bearer tokens prove that whoever holds the token may access an API; they prove nothing about what the holder intended to do, who delegated the action, or whether the holder's runtime state is still aligned with the original authorization. Vouch credentials bind identity, intent, target, and resource together, making non-repudiable attestation possible.

iii.
Is Vouch Protocol production-ready?

Yes for the credential layer (signing, verification, delegation chains, hybrid post-quantum profile, three-language interop). The State Verifiability layer (Heartbeat Protocol orchestration, validator quorum, canary commitments) is informative in this revision and expected to become normative as implementer experience accumulates.

For Developers

Integration & SDKs

iv.
Which languages have Vouch SDKs?

Three: Python (pip install vouch-protocol), TypeScript (npm install @vouch-protocol/sdk), and Go (go install github.com/vouch-protocol/vouch/go-sidecar/cmd/vouch-sidecar). All three produce byte-identical credentials thanks to RFC 8785 JCS canonicalization. Cross-language test vectors are published.

v.
Do I need to run the Go sidecar?

Only if you want the Identity Sidecar pattern, which is the recommended deployment for LLM-driven agents because it keeps the private key out of the LLM context window. If your code never exposes the key to the model, you can sign directly from the Python or TypeScript SDK.
